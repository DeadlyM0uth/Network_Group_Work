**Автор:** Смирнов Вадим Андреевич гр 4150
**Цель:** Исследовать модели доступа к комнатам (public/private), видимость списков комнат, историю сообщений, поведение неавторизованных пользователей, последствия выхода/бана и связанные аспекты безопасности и приватности в развернутой инстанции Synapse.

### 1. Методология

Для исследования был развернут экземпляр Synapse (с стандартной конфигурацией `homeserver.yaml`, federation включен, но с рекомендуемыми ограничениями для приватности). Тестирование проводилось с использованием клиентов Element Web/Desktop, нескольких аккаунтов (администратор, обычный пользователь, гость, неавторизованный), а также инструментов API (Admin API, Client-Server API). Проверялись сценарии на локальном сервере и через federation. Конфигурация включала `allow_public_rooms_without_auth: false` и `allow_public_rooms_over_federation: false` по умолчанию.

### 2. Публичные и приватные комнаты: доступ и видимость списков

- **Может ли неавторизованный пользователь видеть список комнат?**  
  Нет (в стандартной конфигурации). Доступ к публичному списку комнат (`/publicRooms`) требует аутентификации, если `allow_public_rooms_without_auth` установлен в `false` (значение по умолчанию). Неавторизованные пользователи (без access token) не могут получить список публичных комнат сервера.

- **Может ли обычный пользователь видеть чужие комнаты?**  
  Обычный пользователь видит только те комнаты, в которых состоит, или публичные комнаты (если они опубликованы в директорию и он имеет доступ к списку). Чужие приватные (invite-only) комнаты не видны, если нет приглашения или знания alias/ID. Пользователи не могут просматривать полный список всех комнат на сервере (это доступно только администраторам через Admin API).

- **Кто может создавать публичные комнаты?**  
  По умолчанию любой зарегистрированный пользователь может создавать комнаты и публиковать их в публичную директорию. Администратор сервера может ограничить это через `room_list_publication_rules` в конфигурации Synapse.

### 3. История сообщений и видимость для новых участников

- **Видит ли новый участник старую историю?**  
  Зависит от настройки `history_visibility` комнаты:
  - `joined` (по умолчанию): Новый участник видит только сообщения после своего присоединения.
  - `invited`: Видит сообщения после получения приглашения.
  - `shared`: Видит историю до момента изменения настройки.
  - `world_readable`: История доступна всем.

  В E2EE-комнатах новые участники по умолчанию не видят старые зашифрованные сообщения без обмена ключами.

### 4. Поведение после выхода из комнаты

- **Что происходит после выхода из комнаты?**  
  Пользователь удаляется из списка участников. Комната исчезает из списка активных в клиенте, но история сохраняется на сервере и у других участников.

- **Можно ли открыть старые media-ссылки после выхода?**  
  Да, если media не удалена с сервера (mxc:// URL хранятся независимо). Ссылки работают, пока файлы не очищены по retention policy.

### 5. Поведение после удаления/бана

- **Можно ли писать в комнату после удаления/бана?**  
  Нет. Бан предотвращает повторное присоединение и отправку сообщений. Ban сохраняется до unban.
